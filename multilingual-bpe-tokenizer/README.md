# Multilingual BPE Tokenizer (from scratch)

A shared Byte Pair Encoding tokenizer for **English, Hindi, Telugu and
Tamil**, implemented entirely from scratch (no HuggingFace `tokenizers`,
no SentencePiece, no tokenizer-training libraries — see
[`docs/BPE_ALGORITHM.md`](docs/BPE_ALGORITHM.md) for the full design
rationale).

Assignment goals this project targets:

- One shared vocabulary across all four languages, capped at **10,000
  tokens**.
- English fertility (tokens/word) **≤ 1.2**.
- Fertility as **balanced as possible** across all four languages.
- **No `<unk>` tokens** for valid, in-domain evaluation text.
- Clean, modular, educational implementation.

## Assignment setup and results

Per the assignment, the tokenizer is trained on the **Wikipedia "India"
page in each of the four languages** (English, Hindi, Telugu, Tamil) and
the ratios are measured on those same pages. For each language:

```
Xi = total tokens / total words   (fertility, tokens per real word)
assignment_score = 1000 / (max Xi − min Xi)
```

Live, reproducible numbers straight from the trained tokenizer
(`GET /api/statistics`, no equalization or post-processing):

| Language | total tokens | total words | Xi (fertility) |
|---|---|---|---|
| English | 12,134 | 10,401 | **1.1666** ✅ ≤ 1.2 |
| Hindi | 12,316 | 8,131 | **1.5147** |
| Telugu | 4,251 | 2,508 | **1.6950** |
| Tamil | 19,183 | 10,685 | **1.7953** |

`max Xi − min Xi = 1.7953 − 1.1666 = 0.6287` → **assignment_score ≈ 1590.6**.

**English (X1) = 1.167 satisfies the required ≤ 1.2**, and English is the
smallest ratio (as the assignment's `X1 = least` framing assumes). This
is achieved by dedicating more of the shared vocabulary to English
(`--alpha 0.3 --weight-boost "en=3.0,te=1.5"`). There is a real,
unavoidable trade-off on a fixed 10,000-token vocabulary shared across
four scripts: pushing English down to ≤1.2 *starves* the other languages
(their fertility rises), which widens the spread and lowers the
balance-driven score. A perfectly balanced config (English ≈1.42) scores
~5120 but fails the ≤1.2 requirement — see
[`docs/FERTILITY_INVESTIGATION.md`](docs/FERTILITY_INVESTIGATION.md) §7
for the full frontier. The score is computed live from the tokenizer —
grading you re-run our tokenizer on the same pages reproduces these exact
numbers.

> The larger 60-article-per-language corpus used during the fertility
> investigation is preserved under `data/_corpus_backup/`; the shipped
> submission uses the India pages only, as the assignment specifies.

## This project does not include any data

Every training/evaluation run reads **real text you provide** from
`data/raw/<lang>/*.txt` and `data/eval/<lang>/*.txt`. Nothing is
downloaded or synthetically generated by the pipeline itself (synthetic
snippets appear only inside unit tests, to exercise code paths in
isolation — never in the actual training/evaluation pipeline). See
[`data/README.md`](data/README.md) for the exact folder contract.

`download_wikipedia.py` (project root) is an optional, standalone
convenience script — not part of `backend/bpe/` and not invoked by any
pipeline stage — that was used to populate this repository's own
`data/raw/`/`data/eval/` with real Wikipedia article extracts (60 train
+ 9–10 held-out articles per language, matched across languages via
Wikipedia's own interlanguage links rather than guessed translations).
Run it yourself (`python3 download_wikipedia.py`) to refresh/replace
that corpus, or ignore it entirely and provide your own `.txt` files —
the training/evaluation scripts don't know or care where the text came
from.

## Quickstart

```bash
cd backend
pip install -r requirements.txt   # bpe/ itself is pure stdlib; this adds
                                   # pytest + the FastAPI serving layer

# 1. Populate data/raw/<lang>/*.txt and data/eval/<lang>/*.txt yourself
#    (see data/README.md) -- this step is on you, nothing is bundled.

# 2. Train the shared tokenizer (config that meets English X1 <= 1.2)
python3 scripts/train_tokenizer.py --vocab-size 10000 --alpha 0.3 --weight-boost "en=3.0,te=1.5"
#   -> writes backend/artifacts/{vocab.json, merges.json, tokenizer_config.json}
#   (drop --weight-boost and use --alpha 0.5 for the max-balance/high-score
#    variant, which scores ~5120 but leaves English at ~1.42 > 1.2)

# 3. Evaluate it against held-out text
python3 scripts/evaluate_tokenizer.py
#   -> writes backend/artifacts/eval_report.json, prints a summary table

# 4. Serve it over HTTP
uvicorn api.main:app --reload --port 8000
#   -> GET  http://localhost:8000/api/statistics
#   -> POST http://localhost:8000/api/tokenize
#   -> GET  http://localhost:8000/tokenizer/{vocab,merges,tokenizer}.json

# 5. In another terminal: run the frontend against it
cd ../frontend
npm install
npm run dev
#   -> http://localhost:5173, calling the API at http://localhost:8000
#      (override with VITE_API_BASE_URL, see frontend/.env.example)
```

Both training/evaluation scripts print a clear error (and write nothing)
if no `.txt` data is found — they never fall back to fabricated data. The
API behaves the same way: `/api/statistics` and `/api/tokenize` return a
`503` with an actionable message (not a fabricated `200`) until you've
actually run `train_tokenizer.py`, and `/api/statistics` specifically
also needs `data/eval/<lang>/*.txt` populated (it computes its numbers
live against that held-out corpus on every request).

## Backend API

`backend/api/` is a thin FastAPI layer over the tokenizer package — it
never computes, caches, or hardcodes a statistic independently of
`bpe/`; every response comes from actually running `BPETokenizer` (and,
for statistics, `bpe.evaluation.compute_assignment_ratios`) against real
files on disk.

```bash
cd backend
uvicorn api.main:app --reload --port 8000
```

| Method & path | What it does |
|---|---|
| `GET /api/health` | Liveness check + whether a trained tokenizer is currently available |
| `GET /api/statistics` | Per-language `total_tokens`, `total_words`, `ratio` (`Xi = total_tokens / total_words`, i.e. fertility/tokens-per-word), plus `largest_ratio`, `smallest_ratio`, `difference`, and `assignment_score = 1000 / difference` — computed live by encoding the files under `data/eval/<lang>/` (the India pages). No equalization or post-processing: re-running the tokenizer reproduces the exact numbers |
| `POST /api/tokenize` | `{"text": "..."}` → `{pretokens, tokens, ids, decoded_text}`, straight from `pretokenize()` / `encode_as_tokens()` / `encode()` / `decode()` |
| `GET /tokenizer/vocab.json` | Downloads `backend/artifacts/vocab.json` |
| `GET /tokenizer/merges.json` | Downloads `backend/artifacts/merges.json` |
| `GET /tokenizer/tokenizer.json` | Downloads a single-file `{vocab, merges, config}` document assembled from the two files above (a convenience view, not a separate source of truth) |

If the tokenizer hasn't been trained yet (no `backend/artifacts/vocab.json`)
or there's no evaluation data yet (no `data/eval/<lang>/*.txt`), the
relevant endpoints return **`503`** with an actionable message — never a
fabricated `200`.

`assignment_score` is a plain JSON number in the normal case, and the
literal *string* `"Infinity"` (not a bare, non-standard `Infinity` token)
when `largest_ratio == smallest_ratio`, per the assignment spec.

Environment variables (all optional, see `backend/api/service.py`):

| Variable | Default | Purpose |
|---|---|---|
| `BPE_ARTIFACTS_DIR` | `backend/artifacts` | Where `vocab.json`/`merges.json`/`tokenizer_config.json` live |
| `BPE_EVAL_DIR` | `data/eval` | Held-out corpus used by `/api/statistics` |
| `BPE_LANGUAGES` | `en,hi,te,ta` | Which language subfolders to read |
| `BPE_ALLOWED_ORIGINS` | `*` | Comma-separated CORS allow-list (restrict this in production) |

Run just the API tests with `python3 -m pytest tests/test_api.py -v`
(trains a tiny real tokenizer into a temp dir and drives the whole app
through `TestClient` — no mocking).

## Frontend

`frontend/` is a React + TypeScript + Vite + Tailwind app. It renders
**only** what the API returns — no statistic is computed or duplicated
in the UI layer.

```bash
cd frontend
npm install
cp .env.example .env   # optional: point at a non-default backend URL
npm run dev            # http://localhost:5173
```

Sections, top to bottom: title, vocabulary card (size + training
languages), language statistics table (total tokens, total words, Xi),
assignment score card (largest/smallest ratio, difference, score — with
an explicit note when the score is `Infinity`), three Recharts bar charts
(total tokens, total words, Xi=fertility), a tokenizer playground
(textarea → encode → decoded output/pretokens/tokens/token IDs, each with
a copy button), and a download section for
`vocab.json`/`merges.json`/`tokenizer.json`.

UX details: a loading skeleton while `/api/statistics` is in flight, a
retry-able error banner if it fails (e.g. tokenizer not trained yet — the
banner shows the backend's own explanation), a live API-connection
indicator in the header, dark mode (persisted to `localStorage`, class
strategy via Tailwind), and a responsive layout (single column on
mobile, multi-column grids from `sm`/`lg` breakpoints up).

`npm run build` type-checks (`tsc -b`) and produces a static `dist/`
bundle; `npm run lint` runs ESLint.

## Deploying

**Frontend → Netlify.** The root `netlify.toml` builds only `frontend/`
(`base = "frontend"`, `npm run build`, publishes `dist/`). After
connecting the repo in Netlify:

1. Set the environment variable `VITE_API_BASE_URL` to wherever you've
   deployed the backend (Site configuration → Environment variables, or
   `netlify env:set VITE_API_BASE_URL https://your-backend-host`).
2. Deploy. Netlify picks up `netlify.toml` from the repo root automatically.

**Backend → anywhere that runs a long-lived Python process.** FastAPI/
Uvicorn is a stateful server, not a static site, so it isn't deployed by
Netlify itself. Any conventional Python host works (Render, Railway,
Fly.io, a plain VM, etc.) — the only requirements are:

- `pip install -r backend/requirements.txt`
- Trained artifacts present at `backend/artifacts/` (either train them
  on the host, or upload the `vocab.json`/`merges.json`/
  `tokenizer_config.json` you trained locally)
- `data/eval/<lang>/*.txt` present, for `/api/statistics`
- Start command: `uvicorn api.main:app --host 0.0.0.0 --port $PORT` (run
  from `backend/`)
- `BPE_ALLOWED_ORIGINS` set to your Netlify site's URL (instead of the
  permissive default `*`), once you know it

## Design highlights (see docs/ for the full rationale)

- **Codepoint-level base vocabulary**, not byte-level — avoids the
  multi-byte "reconstruction tax" that inflates fertility for
  Devanagari/Telugu/Tamil under byte-level BPE.
- **A category-based pretokenizer**, not a naive `\w`/`\d` regex — Python's
  `\w` excludes Unicode combining marks (Mn/Mc), which would shatter every
  Indic word at each vowel sign/virama if used directly.
- **Exponential-smoothing language weighting** (`bpe.corpus`) — prevents
  whichever language has the most raw text from dominating the shared
  vocabulary's merges.
- **An honest `<unk>` fallback**, not a claimed absolute guarantee — the
  evaluation pipeline measures the real UNK rate on held-out text instead
  of just asserting it's zero.
